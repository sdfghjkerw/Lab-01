# Lab 9.1: News Aggregator (Observer & Pub/Sub Patterns)

## Сравнение: Observer vs Pub/Sub
* **Observer (Наблюдатель):** Издатель напрямую хранит список подписчиков. Существует прямая зависимость (coupling).
* **Pub/Sub (Издатель-Подписчик):** Издатель и Подписчик не знают друг о друге. Они общаются через посредника — **EventBus**. Это обеспечивает полную "развязку" (decoupling) системы.

## Когда использовать?
* Используйте **Pub/Sub**, когда система становится большой и компоненты должны быть независимы (например, разные модули сайта не должны зависеть от логики уведомлений).
* Используйте **Observer**, если вам нужно простое оповещение внутри одного объекта.

## Управление памятью (Memory Management)
В данной работе реализован механизм **Cleanup**:
1.  Метод `subscribe` возвращает функцию-отписку.
2.  При уничтожении подписчика вызывается `unsubscribe()`, что удаляет ссылки на колбэки из `EventBus`.
3.  Это предотвращает утечки памяти (memory leaks), так как сборщик мусора (GC) может спокойно удалить неиспользуемые объекты.
