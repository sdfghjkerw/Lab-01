data validation function 
проблема - нужна проверка формата из за возможности получения некорректонго имейла или телефона
контектс - для любых форм регистрации авторизации, проверки пароля номера почты, обратная связь, получение информации на почту
решение - фнукция принимает значение что вводят в инпут и проверяет ее, пропускает через логику проверки совпадения 
компоненты - входные данные, результаты true false, путсеы значения

pattern name - emiael validation function
description - паттерн проверяет точность совпадения email адреса, проверяет указан ли домен, чтобы избежать ошибки при вводе
context outline - для любых форм регистрации авторизации, проверки пароля номера почты, обратная связь, получение информации на почту, для любых форм где требуется подтверждение имейла
problem statement - неверный ввод домена, приводящий к невозможности связаться с пользователями после
solution - полученный через инпут ввод пароля, проверяется по регуряному выраэению, и если он верный получаем true, если нет - false, дальше пользователю выводится либо успешный ввод, либо указание на ошибку
design - вход по строке инпута email, проверка - по регулярному выражению, выход - нбулевое значение(true, false), поведение - незаметно для пользователя 

examples - регистрация нового пользователя, проверка уже вошедшего пользовател, проверка почты для рассылки, подтверждение личности
consequences - из плюсоы - проверка почты, проверка верного ввода домена почты и его присутствие. минусы - нет проверки что почта точно существует
corequisites - singleton pettern - модуль для проверки на единичный экземпляр модуля, factory-pattern - модуль для создания связанных объектов и предоставления разных вариантов реализации

relations - revealing-module-pattern модуль указывает публичные методы и свойства, а приватные нет
namespace-pattern - для решения проблемы глобального пространства имен, но не так структурировано

discussions - сичтаю что это важный паттерн для проверки, важный - для точной обратной связи, верного входа на плафторму/сайт/приложение и тд. логика легкая, взаимодействует не с локальными данными а с инпутами
на основе его можно проверять не только имейл но и ввод какх-либо хдругих требующиз определенных значений.
