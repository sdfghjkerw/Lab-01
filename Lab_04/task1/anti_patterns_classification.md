1. global namespace pollution 
неблагоприятная ситуация: объявление множетсва переменных UserName, config, util непосредственно в глобальной области видимости. это создает риск конфликтов имен когда разные части кда или сторонние библиотеки могут случайно перезаписать данные друг друга
хорошее решение:использование Namespace Pattern с 11 главы, все переменные группируются в одном глобальном объект-контейнере.
2. tight coupling 
неблагоприятная ситуация: функции напрямую обращаются к глобальным переменным и зависят от вызова других глобальных функций. это делает код хрупким и невозможным для модульного тестирования
хорошее решение: использвоать module pattern из 7 главы . инкапсуляция логикик внутри IIFE, создание приватныз переменных и экспорт только необходимых методов через публичный API
3. String-based function invocation
неблагоприятная ситуация: передача строк вместо функция в setTimeout и setInterval, приводит к скрытому использованию eval() что замедляет работу и небезапасно
хорошее решение: передача анонимынх функций или ссылок на них. вместо setTimeout("updateTimer", 1000) , ипользовать setTimeout(updateTimer, 1000)
4. Prototype Modification
неблагоприятная ситуация: расширение втсроенного объекта object.prototype, что влияет на все объекты в javascript и может привести к непредсказуемому поведению циклов в поломке сторонних библиотек
хорошее решение: использование utility modules, создание отдельного модуля formatter или utils внутри вашего пространтсва имен для  выполнения таких задач
5. Spaghetti Code
неблагоприятная ситуация: отсутствие оргинзациии, где бизнес-логика, работа с данными и манипуляции с dom перемешаны в одном файле
хорошее решение: комбинация module-pattern, namespace pattern, разделение приложения на логические модули, которые организованы в иерархическую структуру внутри общего пространства имен
