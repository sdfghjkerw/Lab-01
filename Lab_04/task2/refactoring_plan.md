////////////////выявленные антипаттерны для утсранения:
1. загрязнение глобального пространства имен - все переменные cart, user, products, и функции находятся в глобальной области видимости
2. сильная связанность - функции напрямую изменяют глобаьное состояние и вызывают друг друга без абстракции
3. дублирование кода - логика валидации и расчетов повторяется в несокльких функциях
4. отсутствие инкапсуляции - внутренние структуры данных открыты и могут быть изменены любым скриптом

///////////////предлагаемые паттерны проектирования
A. namespace pattern - для устранения глобального загрязнения будет создан единый глобальный объект myApp. весь функционал будет организован в логическую иерархию:
- myApp config для налоговых ставок, валюты, натсроек
- myApp utils для формативания и общей логики
- myApp modules для основной логики (корзины и товаров)

B. Module pattern - каждая функциональная область будет обернута в IIFE :
- инкапсуляция - все внутренние массивы(cartItems) станут приватными переменными
- публичный API - только необходимые методы например addItem, init будут экспортированы через возвращаемый объект
- управление состоянием - предотвращает случайное изменение данных извне модуля

/////////////////этапы реализации
1. инициализация пространства имен
2. ивлечение конфигурации
3. создание модуля утилит
4. рефакторинг основной логики
5. основная точка входа

/////////////ожидаемые преимущества
- поддерживаемость - легче обновлять конкрентные модули, не ломая всю систему
- масштабируемость - новые функции могутбыть добавлены как новые модули в пространстве имен myApp
- безопасность - приватная область несанкционированный доступ к данным юзеров или заказов
